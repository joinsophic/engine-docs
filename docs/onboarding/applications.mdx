title: Applications

An **onboarding application** (`onba_...`) ties [persons](/docs/onboarding/persons), consents, and evidence together into one request that kicks off compliance review.

Submission is synchronous. We validate against your tenant's requirements and either accept (`pending`, awaiting review) or return `400` with a precise explanation.

## Request structure

`POST /onboarding-applications` with four top-level fields.

| Field | Type | Required | Description |
|---|---|---|---|
| `customer_type` | enum | yes | `individual`, `business`, or `joint`. See [requirements by customer type](#requirements-by-customer-type). |
| `authorities` | array | yes | Non-empty. Persons acting on the resulting customer, and in what capacity. |
| `consents` | array | no | Legal document consents. Defaults to empty. |
| `evidence` | array | no | Evidence files. Defaults to empty. |

Items reference persons via `person` (not `person_id`).

### Authorities

Each authority pins a person to a role. Valid `authority_type` values: `owner`, `joint_owner`, `signatory`, `power_of_attorney`, `trading`. The accepted set and counts are fixed per `customer_type` — see [authority composition](#authority-composition).

A business application pairs a legal-person `owner` with one or more natural-person `signatory` authorities:

```json
{
  "authorities": [
    {"person": "per_legal_entity", "authority_type": "owner"},
    {"person": "per_ada",          "authority_type": "signatory"}
  ]
}
```

<Note>
A person can hold at most one authority slot. Duplicate `person` IDs across `authorities` return `400 invalid_request`.
</Note>

<Tip>
The duplicate-person check runs before the authority-type check. If the same `person` appears under two `authority_type`s, you'll see the duplicate error first. Use distinct persons per slot when debugging authority-type errors.
</Tip>

### Consents

A consent records that a person agreed to a specific [legal document version](/api-reference/list-versions-of-a-legal-document). See [required consents](#required-consents).

| Field | Type | Required | Notes |
|---|---|---|---|
| `person` | string | yes | An authority or related person — see [attribution](#per-person-attribution). |
| `consent_type` | string | yes | E.g. `terms_of_service`. Must match a required consent for the customer type. |
| `legal_document_version` | string | yes | ID of a [legal document version](/api-reference/list-versions-of-a-legal-document). |
| `consented_at` | datetime | yes | ISO-8601 timestamp. |

<Warning>
Every consent needs a valid `legal_document_version`. Missing or unknown IDs fail the application with `400 invalid_request`.
</Warning>

### Evidence

Upload files via the [Files API](/api-reference/create-a-file), then reference the returned IDs.

| Field | Type | Required | Notes |
|---|---|---|---|
| `person` | string | yes | Person this evidence is for. |
| `evidence_type` | enum | yes | See [evidence taxonomy](#evidence-taxonomy). |
| `document_type` | enum \| null | conditional | Required for most types; must be `null` for `selfie`. |
| `file` | string | yes | ID of an uploaded file. |

## Per-person attribution

Every consent and evidence item's `person` must be either an **authority** on this application or a **related person** declared in a `relationships` claim on a legal authority. Anything else returns `400`.

## Duplicate submissions

Applications are de-duplicated by `customer_type` and the set of `(person, authority_type)` pairs in `authorities`. Consents and evidence aren't part of the key.

While an earlier matching application is `pending` or `processing`, a new submission returns `409 conflict`. Once it reaches `completed`, you can resubmit. There's no self-service cancellation — contact support.

## Requirements by customer type

Requirements are defined **per authority slot** — each `(customer_type, authority_type)` pair carries its own required claims, evidence bundles, consents, expected `person_type`, and count. A business `owner` and `signatory` have different sets; neither inherits.

<Note>
These are the current defaults. Requirements are tenant configuration and evolve — don't hard-code them. Drive your integration from the current error responses; breaking changes ship through our release channels.
</Note>

### Authority composition

| `customer_type` | `authority_type` | Expected `person_type` | Count |
|---|---|---|---|
| `individual` | `owner` | `natural` | exactly 1 |
| `business` | `owner` | `legal` | exactly 1 |
| `business` | `signatory` | `natural` | at least 1 (no upper bound) |
| `joint` | `owner` | `natural` | exactly 1 |
| `joint` | `joint_owner` | `natural` | at least 1 (no upper bound) |

Any other `(customer_type, authority_type)` combination returns `400 invalid_request`. See [composition errors](/docs/onboarding/errors#per-authority-composition-errors).

### Individual

One natural-person `owner`.

**Required claims**: `identity`, `nationalities`, `residence`, `tax_residencies`, `fatca_status`, `pep_status`, `employment`, `financial_profile`, `investment_experience`, `expected_activity`.

**Required evidence** — three independent requirements:

| For the `identity` claim | Acceptable bundles |
|---|---|
| | A single `identity_document` of type `passport` |
| | **or** two `identity_document` items: one `national_id_front` **and** one `national_id_back` |

| For the `identity` claim | Acceptable bundles |
|---|---|
| | A single `selfie` (no `document_type`) |

| For the `residence` claim | Acceptable bundles |
|---|---|
| | One `proof_of_address` with any compatible document type (see [evidence taxonomy](#evidence-taxonomy)) |

**Consents**: owner supplies the standard set (see [required consents](#required-consents)).

### Business

One legal-person `owner`, at least one natural-person `signatory`, plus related persons on the legal owner's `relationships` claim. Owner and signatory slots have different requirements.

#### Owner (legal person)

**Required claims**: `registration`, `contact_details`, `relationships`.

**Required evidence**:

| Claim | Acceptable bundles |
|---|---|
| On `registration` | One `certificate_of_good_standing` |

**Consents**: the legal owner supplies the standard set. Signatories don't consent personally.

**Related persons**: see [related-person requirements](#related-person-requirements).

#### Signatory (natural person)

One per person signing for the business. Signatories don't need legal-entity claims.

**Required claims**: `identity`, `nationalities`, `residence`, `pep_status`.

**Required evidence**:

| For the `identity` claim | Acceptable bundles |
|---|---|
| | A single `identity_document` of type `passport` |
| | **or** two `identity_document` items: one `national_id_front` **and** one `national_id_back` |

| For the `identity` claim | Acceptable bundles |
|---|---|
| | A single `selfie` (no `document_type`) |

No `proof_of_address` needed. Consents are covered by the legal owner.

### Joint

One natural-person `owner` plus one or more natural-person `joint_owner`s. Every party carries the same requirements as an individual owner, each attributed to that person.

**Required claims (per party)**: `identity`, `nationalities`, `residence`, `tax_residencies`, `fatca_status`, `pep_status`, `employment`, `financial_profile`, `investment_experience`, `expected_activity`.

**Required evidence (per party)** — three independent requirements:

| For the `identity` claim | Acceptable bundles |
|---|---|
| | A single `identity_document` of type `passport` |
| | **or** two `identity_document` items: one `national_id_front` **and** one `national_id_back` |

| For the `identity` claim | Acceptable bundles |
|---|---|
| | A single `selfie` (no `document_type`) |

| For the `residence` claim | Acceptable bundles |
|---|---|
| | One `proof_of_address` with any compatible document type |

**Consents**: every natural authority supplies the standard set, each attributed to the party giving it.

### Required consents

Who supplies the consents depends on customer type:

- **Individual** — the owner.
- **Joint** — every natural authority, each attributed to themselves.
- **Business** — the legal owner. Signatories don't consent personally.

<Note>
These are the current defaults. The required set is tenant configuration and may change — drive your integration from error responses.
</Note>

| `consent_type` | What the customer agrees to |
|---|---|
| `terms_of_service` | Your terms of service. |
| `risk_disclosures` | Applicable investor risk disclosures. |
| `data_accuracy` | A statement that the information provided is accurate. |

Each consent needs a live `legal_document_version` — list them via [`GET /legal-documents/{legal_document_id}/versions`](/api-reference/list-versions-of-a-legal-document).

## Related-person requirements

Persons declared in a legal authority's `relationships` claim carry their own requirements per `relationship_type`.

<Note>
These are current defaults and evolve with compliance configuration — drive your integration from error responses.
</Note>

### `director`

**Required claims**: `identity`, `nationalities`.

**Required evidence**:

| For the `identity` claim | Acceptable bundles |
|---|---|
| | A single `identity_document` of type `passport` |
| | **or** two `identity_document` items: one `national_id_front` **and** one `national_id_back` |

| For the `identity` claim | Acceptable bundles |
|---|---|
| | A single `selfie` (no `document_type`) |

No consents required from related persons.

### Other relationship types

`shareholder`, `beneficial_owner`, `trustee`, `settlor`, and `authorised_signatory` aren't enforced at submission today — they're captured on the application, but any follow-up checks happen in compliance processing.

<Note>
Unenforced relationship types may become enforced as compliance configuration evolves. Don't assume today's behaviour is permanent.
</Note>

## Evidence taxonomy

Combinations outside this matrix return `400 invalid_request`.

| `evidence_type` | Allowed `document_type` values |
|---|---|
| `identity_document` | `passport`, `national_id_front`, `national_id_back`, `driving_license`, `driving_license_front`, `driving_license_back` |
| `proof_of_address` | `utility_bill`, `bank_statement`, `tax_document` |
| `selfie` | *(none — `document_type` must be `null`)* |
| `certificate_of_shareholders` | `shareholder_register`, `share_certificate` |
| `certificate_of_directors` | `director_register`, `board_minutes` |
| `certificate_of_good_standing` | `certificate_of_good_standing`, `incumbency_certificate` |
| `financial_statements` | `audited_accounts`, `management_accounts` |
| `ownership_structure` | `ubo_declaration`, `org_chart` |
| `proof_of_registered_address` | `utility_bill`, `bank_statement` |
| `proof_of_operating_address` | `utility_bill`, `bank_statement` |
| `nominee_agreement` | `trust_deed`, `nominee_agreement` |
| `board_resolution` | `board_resolution` |

<Note>
National IDs always use the `national_id_front` + `national_id_back` pair — no single-sided `national_id`. Driving licences accept either the single-sided `driving_license` or the front/back pair.
</Note>

### Which evidence types apply to each person type

Natural-person evidence (`identity_document`, `proof_of_address`, `selfie`) is only valid against natural-person authorities and related persons. Legal-entity evidence (`certificate_of_*`, `financial_statements`, `ownership_structure`, `proof_of_registered_address`, `proof_of_operating_address`, `nominee_agreement`, `board_resolution`) is only valid against legal-person authorities. Mismatches return `400`.

## Response

`201 Created` with the application resource:

```json
{
  "id": "onba_m4k7r9s2",
  "created_at": "2026-04-18T09:30:00Z",
  "modified_at": "2026-04-18T09:30:00Z",
  "customer_type": "individual",
  "status": "pending",
  "outcome": null,
  "application": {
    "customer_type": "individual",
    "authorities": [
      {"person": "per_ada", "authority_type": "owner"}
    ],
    "consents": [
      {"person": "per_ada", "consent_type": "terms_of_service", "legal_document_version": "legal_doc_ver_tos_v3", "consented_at": "2026-04-18T09:30:00Z"}
    ],
    "evidence": [
      {"person": "per_ada", "evidence_type": "identity_document", "document_type": "passport", "file": "fil_passport"}
    ]
  }
}
```

| Field | Description |
|---|---|
| `id` | Application ID, prefix `onba_`. |
| `customer_type` | Echoes the submitted value. |
| `status` | Always `pending` on submission. |
| `outcome` | Always `null` on submission. Populated once terminal. |
| `application` | The submitted payload, stored as-is. |

### Application statuses and outcomes

| `status` | Meaning |
|---|---|
| `pending` | Queued for compliance review. |
| `processing` | Under active review. |
| `completed` | Terminal. Check `outcome`. |

| `outcome` | Meaning |
|---|---|
| `approved` | Approved — we provision the customer and accounts automatically. |
| `rejected` | Rejected. |
| `withdrawn` | Withdrawn before a decision. |

<Note>
Post-submission transitions happen in our compliance systems. Poll the application — webhooks aren't emitted for these yet.
</Note>

## Errors

Most integration failures are `400 invalid_request` with a precise `detail` pointing at the person, claim, consent, or evidence at fault. See [errors](/docs/onboarding/errors) for the catalogue.
