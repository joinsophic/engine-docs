A **claim** (`clm_...`) is a single typed data point attached to a [person](/docs/onboarding/persons) — something like an identity, a residence address, a tax residency, or a set of ownership relationships. Every claim has a `claim_type`, a `status`, and a type-specific `data` payload validated against a strict schema.

Claims are what onboarding applications validate against. A person is essentially the sum of their claims.

## Lifecycle

Every claim moves through up to three statuses:

| Status | Description |
|---|---|
| `pending` | You've just submitted the claim. It's recorded but not yet confirmed by compliance. |
| `verified` | Compliance has confirmed the claim. `verified_at` is populated. |
| `superseded` | A newer claim of the same type replaced it. `superseded_at` is populated. |

### State transitions

```
pending ──▶ verified ──▶ superseded
   │
   └──▶ (deleted)
```

| From | To | Trigger |
|---|---|---|
| *(none)* | `pending` | You create the claim via [`POST /persons`](/api-reference/create-a-person) or [`POST /persons/{person_id}/claims`](/api-reference/create-a-claim-for-a-person). |
| `pending` | *(deleted)* | You call [`DELETE /persons/{person_id}/claims/{claim_id}`](/api-reference/delete-a-pending-claim). Only pending claims can be deleted. |
| `pending` | `verified` | We verify the claim as part of compliance review. This is not self-service. |
| `verified` | `superseded` | You create a newer claim of the same type on the same person. The older verified claim is automatically marked `superseded`. |

<Note>
There is no `pending → superseded` transition. A pending claim can only become `verified` or be deleted. Only `verified` claims get superseded.
</Note>

### Supersession: replacing an already-verified claim

When a customer updates information you've already had verified — say, a new address — submit a fresh claim of the same type. We automatically supersede any prior `verified` claim of that type on the same person in the same request:

```bash
curl -X POST https://api.engine.usesophic.com/persons/per_m4k7r9s2/claims \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{
    "claim": {
      "claim_type": "residence",
      "line1": "42 New Street",
      "postal_code": "EC2A 4RQ",
      "city": "London",
      "country_code": "GB"
    }
  }'
```

After this call:

- The new `residence` claim exists with status `pending`.
- The previously-verified `residence` claim (if any) now has status `superseded` and a `superseded_at` timestamp.
- Any still-pending `residence` claim is left untouched — it's harmless and you can delete it if you don't want it around.

## Creating a claim

To add a claim to an existing person, `POST /persons/{person_id}/claims` with a single `claim` object:

<CodeGroup>
```python Python
import requests

response = requests.post(
    "https://api.engine.usesophic.com/persons/per_m4k7r9s2/claims",
    headers={
        "Authorization": "Bearer YOUR_ACCESS_TOKEN",
        "Content-Type": "application/json",
    },
    json={
        "claim": {
            "claim_type": "pep_status",
            "is_pep": False,
            "has_pep_association": False,
        }
    },
)
claim = response.json()
```

```javascript Node.js
const response = await fetch(

```

## Listing and retrieving claims

List all of a person's claims, optionally filtered:

```bash
curl "https://api.engine.usesophic.com/persons/per_m4k7r9s2/claims?claim_type=residence&status=verified" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN"
```

| Query parameter | Values | Notes |
|---|---|---|
| `claim_type` | Any [`claim_type`](#claim-type-reference) | Optional. Filters by claim type. |
| `status` | `pending`, `verified`, `superseded` | Optional. Omit to return claims of all statuses. |

Retrieve a single claim by ID via [`GET /persons/{person_id}/claims/{claim_id}`](/api-reference/retrieve-a-person-claim).

## Deleting a pending claim

Only `pending` claims can be deleted. Use this to clean up mistakes or abandon a claim before it's verified:

```bash
curl -X DELETE https://api.engine.usesophic.com/persons/per_m4k7r9s2/claims/clm_abc123 \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN"
```

A successful delete returns `204 No Content`. Attempting to delete a `verified` or `superseded` claim returns `400 invalid_request` — those are part of the permanent record and can't be removed through the API.

## Claim type reference

Every claim payload uses `extra: forbid` validation — unknown fields are rejected. Dates are ISO-8601 (`YYYY-MM-DD`), country codes are ISO-3166-1 alpha-2 (`GB`, `US`, `DE`…).

Acceptable when `person_type` is `natural`.

|---|---|---|---|
| `first_name` | string | yes | |

| `date_of_birth` | date | yes | ISO-8601. |

#### `nationalities`

| Field | Type | Required | Notes |
|---|---|---|---|
| `nationalities` | array | yes | Non-empty. Each item: `country_code` (required), `basis` (optional: `birth`, `naturalisation`, `descent`, `marriage`, `investment_programme`, `other`). |

### Shared claim types

Acceptable on both natural and legal persons.

#### `relationships`

Used primarily on legal persons to declare directors, shareholders, UBOs, trustees, settlors, and authorised signatories.

| Field | Type | Required | Notes |
|---|---|---|---|
| `relationships` | array | yes | Non-empty. See entry fields below. |

Each relationship entry:

| Field | Type | Required | Notes |
|---|---|---|---|
| `related_person` | string | yes | ID of an existing person. Must exist — unknown IDs fail the whole request with `400`. |
| `relationship_type` | enum | yes | `director`, `shareholder`, `beneficial_owner`, `trustee`, `settlor`, `authorised_signatory`. |
| `control_type` | enum \| null | no | `ownership`, `voting`, `other`. |
| `ownership_percentage` | decimal (string) \| null | no | E.g. `"25.50"`. |
| `appointment_date` | date \| null | no | |
| `end_date` | date \| null | no | |

<Note>
Different `relationship_type` values carry different follow-on requirements during onboarding. For example, every declared `director` must themselves satisfy an identity / selfie / passport bundle. See [applications — related-person requirements](/docs/onboarding/applications#related-person-requirements).
</Note>

## Reference values

### EUR amount bands

| Value | Range |
|---|---|
| `lt_50k` | Under €50,000 |
| `50k_100k` | €50,000 – €100,000 |
| `100k_250k` | €100,000 – €250,000 |
| `250k_500k` | €250,000 – €500,000 |
| `500k_1m` | €500,000 – €1,000,000 |
| `1m_3m` | €1,000,000 – €3,000,000 |
| `gt_3m` | Over €3,000,000 |

### Industry values

`agriculture`, `mining`, `manufacturing`, `utilities`, `construction`, `wholesale_retail`, `transport_logistics`, `hospitality`, `media_telecoms`, `technology`, `financial_services`, `insurance`, `real_estate`, `professional_services`, `education`, `healthcare`, `arts_entertainment`, `public_administration`, `defence`, `non_profit`, `gambling`, `extractives`, `precious_metals`, `cryptocurrency`, `other`.

Use `other` together with the `_other` text field on whichever claim carries the industry (`employment.industry_other`, `business_nature.business_activity_other`).
