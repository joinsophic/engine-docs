A **person** (`per_...`) is a party we know about — either a natural person (a human) or a legal person (a company, trust, or other legal entity). Persons are standalone: they're not tied to a customer or an account until an [onboarding application](/docs/onboarding/applications) they appear in is approved.

Every person is built from a list of [claims](/docs/onboarding/claims) — typed data points that capture identity, residence, ownership, PEP status, and so on. You can't create an empty person — you have to supply at least one claim.

## Natural vs. legal persons

The `person_type` determines which claim types are acceptable and which evidence a downstream application can request against the person.

| `person_type` | Represents | Example claim types |
|---|---|---|
| `natural` | A human | `identity`, `nationalities`, `residence`, `tax_residencies`, `pep_status`, `employment`, `financial_profile` |
| `legal` | A company, trust, partnership, or other legal entity | `registration`, `contact_details`, `lei`, `registered_address`, `operating_address`, `business_nature`, `relationships` |

A legal person can declare its natural- and legal-person counterparties (directors, shareholders, UBOs, trustees, signatories) using a [`relationships`](/docs/onboarding/claims#relationships) claim. The related persons must already exist as persons of their own — you can't declare a relationship to a party we don't know about.

The split is enforced at the claim layer: every claim type belongs to either natural or legal persons, and mixing them is rejected. Both the `claims` array on `POST /persons` and later calls to `POST /persons/{person_id}/claims` return `400 invalid_request` if a `claim_type` doesn't match the target person's `person_type` — see the [claim type reference](/docs/onboarding/claims#claim-type-reference) for the per-type split.

## Creating a person

Send a `POST` to `/persons` with the `person_type` and a non-empty `claims` array. Creation is **atomic**: if any claim fails validation, nothing is persisted.

<CodeGroup>
```python Python
import requests

response = requests.post(
    "https://api.engine.usesophic.com/persons",
    headers={
        "Authorization": "Bearer YOUR_ACCESS_TOKEN",
        "Content-Type": "application/json",
    },
    json={
        "person_type": "natural",
        "claims": [
            {
                "claim_type": "identity",
                "first_name": "Ada",
                "last_name": "Lovelace",
                "date_of_birth": "1985-12-10",

                "city": "London",
                "country_code": "GB",
            },
        ],
    },
)

```

### Response

A successful request returns `201 Created`:

```json
{
  "id": "per_m4k7r9s2",
  "created_at": "2026-04-18T09:30:00Z",
  "modified_at": "2026-04-18T09:30:00Z",
  "person_type": "natural",
  "full_name": "Ada Lovelace",
  "claims": [
    {
      "id": "clm_abc123",
      "created_at": "2026-04-18T09:30:00Z",
      "modified_at": "2026-04-18T09:30:00Z",
      "claim_type": "identity",
      "status": "pending",
      "data": {
        "first_name": "Ada",
        "last_name": "Lovelace",
        "date_of_birth": "1985-12-10"
      }
    },
    {
      "id": "clm_def456",
      "created_at": "2026-04-18T09:30:00Z",
      "modified_at": "2026-04-18T09:30:00Z",
      "claim_type": "residence",
      "status": "pending",
      "data": {
        "line1": "1 Example Street",
        "postal_code": "SW1A 1AA",
        "city": "London",
        "country_code": "GB"
      }
    }
  ]
}
```

| Field | Description |
|---|---|
| `id` | The person's unique ID, prefix `per_`. |
| `person_type` | `natural` or `legal`. |
| `full_name` | For natural persons: `"{first_name} {last_name}"` derived from the identity claim. For legal persons: the `entity_name` from registration, or empty if none was claimed. |
| `claims` | The claims you provided, now carrying IDs and a `status` of `pending`. See [claim lifecycle](/docs/onboarding/claims#lifecycle). |

<Note>
Persons don't have a status of their own. Lifecycle exists only on individual claims. See the [claims page](/docs/onboarding/claims#lifecycle) for how `pending`, `verified`, and `superseded` work.
</Note>

<Warning>
`POST /persons` doesn't de-duplicate retries. If you retry a request that may have already succeeded, you'll create a duplicate person. Record the returned `id` the first time through and reuse it on retries.
</Warning>

## Retrieving a person

Use [`GET /persons/{id}`](/api-reference/retrieve-a-person) to fetch a person together with its current set of claims:

```bash
curl https://api.engine.usesophic.com/persons/per_m4k7r9s2 \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN"
```

To fetch claims independently — for example, filtered by type or status — use [`GET /persons/{person_id}/claims`](/api-reference/list-claims-for-a-person):

```bash
curl "https://api.engine.usesophic.com/persons/per_m4k7r9s2/claims?status=verified" \
  -H "Authorization: Bearer YOUR_ACCESS_TOKEN"
```

## Authorities

Once an onboarding application that references a person is approved, that person becomes an **authority** on the resulting customer and accounts. Valid authority types are:

| `authority_type` | Meaning |
|---|---|
| `owner` | The primary owner of the customer. |
| `joint_owner` | A joint owner on a shared customer. |
| `signatory` | Can sign on behalf of the customer but doesn't own it. |
| `power_of_attorney` | Acts under a power of attorney arrangement. |
| `trading` | Authorised to place trades on the customer's accounts. |

To list the authorities a given person holds across our platform, use [`GET /persons/{id}/authorities`](/api-reference/list-authorities-for-a-person). The inverse lookup — "who can act on this customer?" — lives at [`GET /customers/{id}/authorities`](/api-reference/list-authorities-for-a-customer).

## Legal persons and relationships

For a business onboarding, create one person for the legal entity and one person for every director, shareholder, UBO, trustee, settlor, or authorised signatory you need to reference. Then add a `relationships` claim on the legal person pointing to each of them.

```json
{
  "person_type": "legal",
  "claims": [
    {
      "claim_type": "registration",
      "legal_name": "Ada & Co Ltd",
      "registration_number": "12345678",
      "date_of_incorporation": "2020-06-01",
      "country_of_incorporation": "GB",
      "entity_type": "private_limited_company"
    },
    {
      "claim_type": "relationships",
      "relationships": [
        {
          "related_person": "per_ada",
          "relationship_type": "director",
          "appointment_date": "2020-06-01"
        },
        {
          "related_person": "per_grace",
          "relationship_type": "shareholder",
          "ownership_percentage": "50.00"
        }
      ]
    }
  ]
}
```

The related persons must already exist. If any `related_person` ID is unknown, the whole request fails with a `400` and nothing is persisted.

See [claim types](/docs/onboarding/claims#relationships) for the full schema of a `relationships` claim.
